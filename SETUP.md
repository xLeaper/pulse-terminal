# Pulse Terminal: setup guide

Start-to-finish: get the app running locally, then wire it to GitHub and Hugging Face.

## Architecture

```
┌──────────────┐     git push       ┌──────────────┐    sync-to-hf.yml    ┌──────────────┐
│  Your laptop │ ─────────────────▶ │   GitHub     │ ──────────────────▶  │   HF Space   │
│   (or CI)    │                    │   (source)   │   force-push mirror  │   (deploy)   │
└──────────────┘                    └──────────────┘                      └──────────────┘
                                          │  daily-refresh.yml,
                                          │  lighthouse.yml,
                                          │  visual-regression.yml,
                                          │  pr-preview.yml
                                          ▼
                                     GitHub Actions
```

**You only edit the GitHub repo.** GitHub Actions push to HF on your behalf. The HF Space is a deploy target, never a source of truth.

## What you need

| | |
|---|---|
| **Node** | 22.x LTS (the Dockerfile pins this) |
| **GitHub account** | with a personal access token *or* `gh` CLI installed |
| **Hugging Face account** | + a write-scope token from https://huggingface.co/settings/tokens |
| **Local repo** | this `app/` directory |

You do **not** need: Docker installed locally (HF builds it), Vercel, or a paid HF tier.

## Step 1 · Local setup

```bash
cd /Users/ly__kevin/Claude/jobpulse-terminal/app

npm install                       # one-time, ~80 packages
node scripts/build-snapshot.mjs   # generates lib/snapshot.json from data/*.csv
npm run dev                       # http://localhost:3000
```

Open localhost:3000 — you should see the Pulse Terminal entry page (italic question, amber cursor). If it loads, your local env is good.

To verify the data pipeline:

```bash
npm run data:refresh
# scrape → enrich → snapshot. Should print:
#   [scrape]   ~250 unique postings
#   [enrich]   ~250 rows
#   [build-snapshot]  N postings · M skills · K categories
```

## Step 2 · Create the GitHub repo

```bash
# If .git already exists pointing at HF, reset it.
rm -rf .git
git init -b main
git add .
git commit -m "Initial commit: Pulse Terminal"

# Create the GitHub repo and push in one shot:
gh repo create pulse-terminal --private --source=. --push

# Or, if you'd rather use the web UI:
#   1. Create empty repo at https://github.com/new (name: pulse-terminal, do NOT init with README)
#   2. git remote add origin https://github.com/<you>/pulse-terminal.git
#   3. git push -u origin main
```

**What ends up in the GitHub repo (everything):**

| Path | What it is |
|---|---|
| `app/`, `components/`, `lib/` | Next.js source |
| `data/*.csv` | Job postings dataset |
| `scripts/` | Scraper, enrichment, snapshot, screenshot, visual-diff |
| `public/` | Static assets |
| `Dockerfile` | The recipe HF uses to build the container |
| `README.md` | HF Space metadata (YAML frontmatter) + project intro |
| `package.json`, `package-lock.json`, `tsconfig.json`, `next.config.ts`, `postcss.config.mjs` | Standard Next config |
| `lighthouserc.json` | Lighthouse thresholds |
| `.github/workflows/*.yml` | The 5 automations |
| `.gitignore`, `.dockerignore` | Ignore rules |
| `screenshots/baseline/*.png` | Visual-regression baselines (ground truth) |
| `SETUP.md`, `SETUP-AUTOMATION.md` | These docs |

**What does NOT get committed (regenerated automatically):**

| Path | Regenerated by |
|---|---|
| `node_modules/` | `npm ci` (Docker `deps` stage) |
| `.next/` | `npm run build` (Docker `builder` stage) |
| `lib/snapshot.json` | `node scripts/build-snapshot.mjs` (Docker + daily refresh) |
| `screenshots/*.png` (non-baseline) | `npm run qa:screenshots` (CI only) |

`.gitignore` already enforces all of this. Verify with:

```bash
git ls-files | wc -l        # should be ~50 files
git ls-files | grep node_   # should be empty
```

## Step 3 · Create the HF Space

You need an empty Space that the GitHub Action will mirror into.

1. Go to https://huggingface.co/new-space
2. Set:

| Field | Value |
|---|---|
| **Owner** | your HF username |
| **Space name** | `pulse-terminal` (or anything; remember the slug) |
| **License** | MIT |
| **SDK** | **Docker** → **Blank** template (do *not* pick Gradio/Streamlit) |
| **Hardware** | CPU basic (free tier) is fine |
| **Visibility** | public or private, your call |

3. Click **Create Space**. You'll land on an empty Space with a starter README.

**Do NOT push anything to the HF Space directly.** The next step makes GitHub do it for you.

What ends up in the HF Space repo (after first GitHub push): an exact mirror of your GitHub repo's `main` branch. The Docker build stages handle everything else — `npm ci`, snapshot generation, `next build`, runtime.

---

## Step 4 · Wire GitHub → HF

### 4a. Get a Hugging Face write token

1. https://huggingface.co/settings/tokens
2. **New token** → name: `pulse-terminal-deploy` → role: **Write**
3. Copy the `hf_…` value (you won't see it again)

### 4b. Add it as a GitHub repo secret

GitHub repo → **Settings** → **Secrets and variables** → **Actions** → **New repository secret**

| Field | Value |
|---|---|
| Name | `HF_TOKEN` |
| Secret | the `hf_…` token |

### 4c. Edit two workflow files

Open `.github/workflows/sync-to-hf.yml` — find the `env:` block:

```yaml
env:
  HF_USERNAME: YOUR_HF_USERNAME      # ← change this
  HF_SPACE:    pulse-terminal        # ← change this if your slug differs
```

Open `.github/workflows/pr-preview.yml` — find:

```yaml
env:
  HF_USERNAME: YOUR_HF_USERNAME      # ← same value as above
  PREVIEW_SLUG_PREFIX: pulse-terminal-pr  # leave as-is unless you want a different naming pattern
```

The other three workflows (`daily-refresh`, `lighthouse`, `visual-regression`) need no editing.

Commit:

```bash
git add .github/workflows/sync-to-hf.yml .github/workflows/pr-preview.yml
git commit -m "Wire HF target Space"
git push
```

## Step 5 · First push + verify

The push above just triggered `sync-to-hf.yml` automatically. Watch it:

1. GitHub repo → **Actions** tab
2. Click the running "Sync to Hugging Face Space" run
3. Wait ~3–5 min — should end green

Then:

1. Go to your HF Space (`https://huggingface.co/spaces/<you>/pulse-terminal`)
2. **Logs** tab — you'll see the Docker build (deps → builder → runner). Takes another 3–5 min on first build.
3. When it says `Ready on http://0.0.0.0:7860`, the Space is live.
4. Visit `https://<you>-pulse-terminal.hf.space` → the Pulse Terminal entry page should render.

If something fails, jump to the [Troubleshooting](#troubleshooting) section.

## Daily flow afterward

```bash
# Make changes — local edits or via Claude Code
git add .
git commit -m "Whatever changed"
git push

# 4 things happen automatically:
#   1. sync-to-hf.yml mirrors to HF
#   2. HF rebuilds the Docker image
#   3. ~5 min later, prod is live
#   4. (If you opened a PR instead, lighthouse + visual-regression + pr-preview also run)

# Once a day at 06:00 UTC:
#   daily-refresh.yml scrapes fresh job postings, commits the new CSVs, triggers another HF build
```

You don't push to HF directly anymore. Everything goes through GitHub.

| `README.md` (HF YAML frontmatter) | ✓ | ✓ (mirror) | — |

The HF Space repo ends up identical to the GitHub `main` branch (HF gets force-pushed). HF then runs the Dockerfile to produce the runtime artifacts.

## What each workflow does (in 1 line)

| Workflow | Triggers on | Purpose |
|---|---|---|
| `sync-to-hf.yml` | push to main | Deploy to prod HF Space |
| `daily-refresh.yml` | cron 06:00 UTC | Scrape fresh data, commit, redeploy |
| `lighthouse.yml` | every PR | Fail if accessibility < 90 |
| `visual-regression.yml` | every PR | Fail if any UI surface drifted >0.5% |
| `pr-preview.yml` | PR open/update/close | Spawn temp HF Space per PR |

Full reference for each: [SETUP-AUTOMATION.md](SETUP-AUTOMATION.md).

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| GitHub Action `sync-to-hf` fails 401 | Token wrong or missing | Check the `HF_TOKEN` secret in GitHub Settings → Secrets |
| GitHub Action `sync-to-hf` fails 403 | Token is read-only | Regenerate at huggingface.co/settings/tokens with **Write** scope |
| GitHub Action `sync-to-hf` fails "not found" | `HF_USERNAME` / `HF_SPACE` wrong | Edit the two env values to match your Space's URL exactly |
| HF build fails `addgroup: gid '1000' in use` | Conflicting user creation | Already fixed in current Dockerfile (uses built-in `node` user) |
| HF build fails `Cannot find module .../next/dist/build/swc.linux-x64-musl.node` | Alpine missing glibc shim | Already fixed — Dockerfile installs `libc6-compat` |
| HF build OOM `JavaScript heap out of memory` | Build container too small | Bump `NODE_OPTIONS=--max-old-space-size=4096` to 6144 in Dockerfile |
| HF builds but shows old code | HF cached aggressively | Space → Settings → **Factory rebuild** |
| Daily refresh "no changes" | Sources returned the same set | Normal — commit only happens when CSVs differ |
| Lighthouse fails on color-contrast | UI changed, contrast dropped | Fix the contrast (open the report URL in the workflow log to see which DOM nodes) |
| Visual regression failed but the change is intentional | Baseline is now stale | Locally: `UPDATE_BASELINE=1 npm run qa:visual-diff` then commit `screenshots/baseline/` |
| PR preview Space won't create | First-time API permission missing | Open https://huggingface.co/settings/spaces, accept the API ToS once |

## What's NOT automated (and where to go next)

- Schema changes to the CSV columns (manual: edit `scripts/enrich.mjs`'s `COLUMNS` array)
- Adding new scrape sources (manual: drop a new function into `scripts/scrape.mjs`)
- Lighthouse threshold tuning (manual: edit `lighthouserc.json`)
- Approving visual diffs (manual by design — pixelmatch can flag legit changes)

For deeper workflow internals, see [SETUP-AUTOMATION.md](SETUP-AUTOMATION.md).
